The problem: queues under pressure

A trading system needs to pass millions of events/sec between threads. The naive solution — a BlockingQueue — works fine at low throughput but falls apart under load. Why?

Watch what happens as producers overwhelm consumers
Three killers: (1) Locks — every put/take acquires a mutex, causing context switches. (2) Allocation — new nodes are created per message, pressuring the GC. (3) False sharing — producer and consumer metadata often share a CPU cache line, causing invisible invalidations.

The ring buffer: pre-allocated, lock-free, cache-friendly

Instead of a dynamic queue, the Disruptor uses a fixed-size circular array of pre-allocated event slots. Producers overwrite slots in place — no allocation, no GC.

Power of 2 sizing: With 1024 slots, index = sequence & 1023 (bitwise AND) replaces sequence % 1024 (division). Division is ~20–80 cycles; bitmasking is 1 cycle. At millions of ops/sec this matters.

Sequencing: one writer, zero contention

The single writer principle: only one thread claims sequence numbers. No CAS loop, no lock. Downstream consumers each track their own cursor independently.

Memory barrier on publish: When the producer calls publish(seq), a store-release barrier ensures the event data is visible to all consumers before the sequence number is. Without this, a consumer could read a sequence number but see stale event data — a memory visibility bug on modern out-of-order CPUs.

Handler pipeline: parallel and sequential stages

Multiple handlers consume the same ring buffer. Dependencies are declared explicitly — journal and replica run in parallel first, then the matching engine runs after both complete. No data is copied between stages.

Diamond dependency: Each handler has its own sequence cursor. The matching engine's barrier depends on both the journal and replica cursors — it will not advance past a slot until both have processed it. All handlers read the same memory — zero copying at any stage.

Wait strategies: trading CPU for latency

When no new events are available, consumers must wait. How they wait is the critical trade-off between latency and CPU consumption.

BusySpin: Consumer loops continuously checking the sequence counter. Detects new events within nanoseconds. Burns 100% of a CPU core. Only viable when that core is isolated and dedicated to this thread.

Disruptor vs BlockingQueue: a direct comparison

Both solve the producer/consumer problem. The difference is in what they sacrifice to get there.

BlockingQueue

Lock on every put and take. New node allocated per message. Head/tail often on same cache line. GC pauses from allocation. Simple API.

Disruptor ring buffer

Single producer: no CAS on write path. Pre-allocated slots: zero allocation. Cache-line-padded sequences. Power-of-2 bitmasking for indexing.

Queue latency profile

p99: ~10–50µs under load. Spikes to milliseconds during GC. Wide, unpredictable tail latency.

Disruptor latency profile

p99: ~200–500ns with BusySpin. p99.9 stays tight — no GC, no lock acquisition, no context switch on the critical path.

When to use a queue

Multiple producers contending. Latency target is milliseconds. Simplicity matters. Bounded memory not required.

When to use Disruptor

Single producer. Sub-microsecond latency target. CPU core can be dedicated. Predictable, bounded memory required.
